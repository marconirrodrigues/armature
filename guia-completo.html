01 — Context

Why this framework exists

Between 2025 and 2026, software engineering underwent an accelerated transformation. AI agents moved beyond glorified autocomplete to become autonomous code producers. But a painful discovery accompanied this advance: building the agent is the easy part — making it work reliably is where the real problem lives.

Three disciplines emerged to solve different facets of this problem:

Spec-Driven Development

Specifications as the source of truth. The spec defines what to build and why, replacing vague prompts with clear contracts between human and agent.

Harness Engineering

The environment surrounding the agent. Constraints, feedback loops, linters, tests — everything that makes an agent operate consistently. Every error becomes a permanent improvement to the system.

Orchestration Engineering

Coordinating multiple specialized agents. Work decomposition, task routing, inter-agent communication — the evolution from pair programmer to team manager.

The problem is that these disciplines evolved in parallel, with different vocabularies, tools, and communities. No one integrated all three into a coherent system.

Central thesis

The three disciplines are not alternatives — they are complementary layers of the same stack. The spec defines the "what". The harness defines "how the agent operates". Orchestration defines "who does what". Using one without the others leaves critical gaps.

The timeline

PeriodParadigmCentral question

This framework integrates the three most recent layers into a progressive approach: you start simple and add complexity only when the project demands it.

02 — The Model

The Progressive 3-Tier Stack

The framework is organized into 3 tiers that activate as needed. Each tier includes the previous ones. It is not necessary to use all three from the start — begin with Tier 1 and evolve when you feel the need.

TIER 3
Orchestration — Scale & Parallelism Multiple specialized agents, task routing, inter-agent communication, cost optimization. For when a single agent is not enough.
TIER 2
Harness — Environment & Reliability Architectural constraints, feedback loops, golden principles, garbage collection, observability. For when the agent needs to operate consistently in production.
TIER 1
Spec-Driven — Intent & Planning Specification, technical plan, decomposed tasks, validation criteria. For any project, from the very first commit.

When to activate each tier

ScenarioTierWhy
Prototype / side projectTier 1Basic spec keeps the agent on track without overhead
Feature in an existing codebaseTier 1Spec forces clarity about how the feature interacts with the system
Ongoing production projectTier 1+2Harness prevents drift, accumulates learning, maintains quality
Legacy modernizationTier 1+2Spec captures lost logic, harness enforces new patterns
System migration (stack, database, cloud, architecture)Tier 1+2MIGRATE mode: maps legacy, defines strategy, ensures rollback and parity
Team with multiple agentsTier 1+2+3Orchestration decomposes and distributes work among agents
Product at scale (e.g.: 1M+ lines)Tier 1+2+3Without orchestration + harness, high throughput = high drift
Practical rule

If you are unsure which tier to use, start with Tier 1. When you feel the agent is "slipping" (ignoring conventions, repeating errors, losing context), it is time for Tier 2. When a single agent becomes a bottleneck, it is time for Tier 3.

03 — Tier 1

Spec-Driven Development

"The spec is documentation that both humans and agents can consult to stay on course. Write good documentation first — the model may be able to build the entire implementation from it."

— Addy Osmani, "How to write a good spec for AI agents"

The philosophy

Instead of code as the source of truth (with specs as disposable accessories), the specification becomes the primary artifact. Code is output generated from specs. Maintaining software means evolving specifications.

The central insight: LLMs are excellent at completing patterns, but poor at reading minds. A detailed spec is simply very good context.

Critical rule: Do not advance to the next phase without human validation of the current phase. Each phase has a review checkpoint.

Phase 1: Specify — What to build

High-level description focused on what and why, not how. User journeys, acceptance criteria, edge cases. The agent can help generate and refine — but the human reviews and approves.

Phase 2: Plan — How to build

Technical constraints: stack, patterns, architecture, dependencies. The agent generates an implementation plan based on the spec. It can generate multiple variations for comparison.

Phase 3: Tasks — Decompose into units

Break the plan into concrete, testable work items. Each task with clear inputs, expected outputs, and validation criteria. Sequenced by dependency. Small enough to implement and test in isolation.

Phase 4: Implement — Execute with verification

The agent executes task by task using spec + plan as context. Atomic commits per task. Validation against criteria defined in the spec.

The 5 principles for good specs

1. Start high, detail later

Write a brief and let the agent expand it into a detailed spec. Do not over-engineer upfront.

2. Structure like a PRD

Clear sections: objective, users, requirements, stack, boundaries. The agent finds information better when it is organized.

3. Modularize the context

Do not dump everything at once. One task at a time. Stacking 10 rules makes the agent follow maybe 3.

4. Include boundaries

What NOT to do is as important as what to do. ✅ Always / ⚠️ Ask first / 🚫 Never.

5. Test and evolve

The spec is a living document, versioned in Git. Incorporate test criteria and update when decisions change.

Warning

"Curse of Instructions": research shows that agents skip the middle of long files, and auto-generated specs worsened results in 5 of 8 tested scenarios (ETH Zurich). Keep specs concise and human-reviewed.

04 — Tier 2

Harness Engineering

"Every time the agent makes a mistake, don't wait for it to improve on its own. Engineer the environment so that specific error becomes structurally impossible to repeat."

— Mitchell Hashimoto, co-founder of HashiCorp

The philosophy

The harness is everything around the agent except the model itself: Agent = Model + Harness. While the spec defines what to build, the harness defines how the agent operates while building it. The metaphor: the model is the horse (fast, powerful, undirected). The harness is the bridle (reins, brake, saddle).

The empirical proof: LangChain went from 52.8% to 66.5% on Terminal Bench 2.0 by changing only the harness, without swapping the model. The model is a commodity — the harness is the moat.

Essential Harness patterns

AGENTS.md as Table of Contents

The OpenAI team started with a giant AGENTS.md — and it failed. The file competed with the code for the model's attention. The solution: ~100 lines working as an index, pointing to a structured docs/ directory. Progressive disclosure: show what is needed at the current level, link to depth when relevant.

Golden Principles

Opinionated, mechanical rules that keep the codebase readable for future agent runs. Encoded in the repo, not in conversations. Enforced mechanically by linters and structural tests — not by "advice" in the prompt.

Garbage Collection

Cleanup agents running on a schedule (daily/weekly): violation scanners, dependency auditors, pattern enforcers. Each deviation becomes a refactoring PR. The OpenAI team spent 20% of their time cleaning up "AI slop" — automating it solved the problem.

Verification beats advice

If a class of error is recurring, stop describing it in AGENTS.md and start preventing it mechanically. Convert repeated review comments into deterministic checks.

Mental model: Guides & Sensors (Böckeler/Fowler)

Guides (feedforward): everything that tells the agent what to do before acting — AGENTS.md, rules, skills, specs. Proactively prevent errors.
Sensors (feedback): everything that verifies the output after — linters, tests, reviews. Reactively detect and correct.
Sensors can be computational (deterministic) or inferential (LLM-based). Use both together.

05 — Tier 3

Orchestration Engineering

"When you orchestrate 50 agents in parallel, vague thinking doesn't just slow you down — it multiplies. A vague spec multiplies errors across the entire fleet. A precise spec multiplies precise implementations everywhere."

— Addy Osmani, "The Code Agent Orchestra"

The philosophy

The evolution of the engineer's role: Coder → Conductor → Orchestrator. The conductor guides a single agent in real time (synchronous, sequential). The orchestrator coordinates a fleet of asynchronous agents, each with its own context, scope, and responsibility.

The mental shift: from pair programming to managing a team. New skills: work decomposition, precise specification, output verification at scale — instead of writing code.

The 3 orchestration patterns

Subagents

The simplest. A parent agent decomposes work and spawns specialized children (Data, Logic, API). No communication between children. The parent manages dependencies manually. Start here.

Agent Teams

The leap. Shared task list with automatic dependency tracking, peer-to-peer messaging, file locking. 3-5 teammates is the sweet spot. Agents self-assign tasks.

Hierarchical

Teams of teams. Feature leads that spawn specialists. Mirrors a real engineering org. The main orchestrator only talks to 2-3 leads, keeping context clean.

The 3 tiers of orchestration tools

TierModeToolsWhen to use
1InteractiveClaude Code CLI, Cursor inlineDay-to-day work, 1 agent
2Local parallelClaude Squad, Conductor, Cursor BG AgentsParallel sprints, 3-10 agents
3Async cloudClaude Code Web, Copilot Agent, Codex WebDraining backlog overnight

Open challenges

Observability is the biggest gap — debugging conflicts between agents is still "whack-a-mole". Cost scales with N agents — the "Plan-and-Execute" pattern (frontier model orchestrates, cheap model executes) reduces it by up to 90%. And spec quality becomes exponentially more critical — ambiguity multiplies across the entire fleet.

06 — File Structure

The practical template

Each tier adds files to the project. All are Markdown, versionable in Git, readable by any agent.

📁 project/ │ ├── SPEC.md ← Tier 1: What to build, for whom, why ├── PLAN.md ← Tier 1: Stack, architecture, technical decisions ├── TASKS.md ← Tier 1: Ordered tasks with criteria ├── MIGRATION.md ← Migrate mode: mapping, strategy, rollback, parity, cutover │ ├── AGENTS.md ← Tier 2: Project Table of Contents (~100 lines) ├── LESSONS.md ← Tier 2: Past errors and how they were resolved ├── PROGRESS.md ← Tier 2+3: Cross-session progress log │ ├── 📁 docs/ │ ├── architecture.md ← Tier 2: Dependency layers, module boundaries │ ├── golden-principles.md ← Tier 2: Mechanical anti-drift rules │ ├── boundaries.md ← Tier 2: ✅ Always / ⚠️ Ask / 🚫 Never │ ├── 📁 patterns/ ← Tier 2: Correct code examples │ │ └── example-api-route.md │ ├── agents-roster.md ← Tier 3: Which agents, roles, scopes │ └── task-routing.md ← Tier 3: How to decompose and distribute work │ └── README.md ← Project overview + quick start
Template design

Each Markdown file contains: (1) a header with purpose and when to use, (2) sections with [DEFINE: ...] placeholders that work as prompts — both for humans to fill in and for agents to use as generative questions, (3) concrete examples between HTML comments, (4) inline instructions explaining the reason for each section.

07 — Implementation

How to use the framework in practice

Full flow — from zero to deploy

Phase 0: Setup (5 minutes)

Copy the files from the tier you need to the root of your repository. Fill in SPEC.md with a high-level brief — even if it is just 3 paragraphs. That is the minimum to get started.

Phase 1: Specify (human + agent)

Open SPEC.md with your agent and ask it to expand the brief into a detailed spec. Review each section. Adjust. Do not advance until you are satisfied with the description of what will be built.

Phase 2: Plan (agent → human review)

With the spec approved, ask the agent to generate PLAN.md. It will propose a stack, architecture, and technical decisions. Question the choices. Ask for alternatives if needed.

Phase 3: Tasks (agent → human review)

The agent breaks the plan into TASKS.md. Each task with an acceptance criterion. Reorder if necessary. Cut scope if too ambitious.

Phase 4: Implement (agent executes, human validates)

The agent works task by task. After each task, validate against the criteria. Atomic commits. If something fails, update LESSONS.md and adjust the harness.

Phase 5: Evolve (continuous)

With every error, improve the harness. With every new feature, update the spec. After every session, update PROGRESS.md. The system improves with each use — compounding effect.

The 4 operating modes

Migrate Mode

Migrate mode has specific discovery steps (M1–M4) to map the legacy system, define a strategy (big bang, strangler fig, parallel execution), and guarantee 3 principles: map before migrating, rollback as a prerequisite, and parity verified — never assumed.

08 — Checklists

Checklists by Tier

Tier 1 — Spec-Driven

  • SPEC.md filled in with objective, users, requirements, and success criteria
  • PLAN.md with stack, architecture, and justified technical decisions
  • TASKS.md with ordered tasks, dependencies, and acceptance criteria
  • Human review between each phase (Specify → Plan → Tasks → Implement)
  • Spec committed to Git alongside the code

Tier 2 — Harness

  • AGENTS.md as table of contents (~100 lines), not an encyclopedia
  • docs/architecture.md with dependency layers and module boundaries
  • docs/golden-principles.md with mechanical rules (not vague advice)
  • docs/boundaries.md with ✅/⚠️/🚫 classification
  • At least 1 example in docs/patterns/
  • Linter + type checker + test suite configured and passing
  • Pre-commit hooks running basic checks
  • LESSONS.md updated after every significant error
  • Constraints enforced mechanically (not just documented)

Tier 3 — Orchestration

  • docs/agents-roster.md defining agents, roles, and file scopes
  • docs/task-routing.md with decomposition criteria
  • PROGRESS.md updated after every work session
  • Each agent operates in an isolated worktree or branch
  • Spec precise enough to survive multiplication by N agents
  • Human review at integration points (when agent outputs converge)
  • Cost tracking per agent/task
09 — Anti-Patterns

What to avoid

Anti-PatternWhy it failsWhat to do instead
AGENTS.md encyclopedia A giant file competes with the code for the model's attention. The agent ignores important constraints. ~100 lines as TOC, pointing to a structured docs/ directory.
Auto-generated spec without review ETH Zurich: auto-generated specs performed worse in 5 of 8 scenarios and cost 20%+ more tokens. Agent generates draft, human reviews and approves before proceeding.
Advice instead of enforcement "Always run the tests" in AGENTS.md is forgotten on the 47th iteration of a long debug session. A pre-commit hook that blocks the commit if tests fail.
Weekly manual cleanup OpenAI spent 20% of their time (every Friday) cleaning up "AI slop". Does not scale. Automated cleanup agents running on a schedule.
Vague spec + multiple agents Ambiguity multiplies by N agents. Each one heads in a different direction. Spec with acceptance criteria, edge cases, and explicit examples before distributing.
New/trendy framework Agents perform better with "boring" technology — composable, stable, well-documented. Favor technologies with strong representation in model training data.
Migrating without mapping the legacy Undocumented system = unmapped risk. Production surprises during cutover cause unplanned rollbacks. MIGRATE mode: discovery M1 maps the legacy before any technical decision.
Migration without a rollback plan Without a defined reversion, any problem becomes an incident. The pressure to "keep going" grows even when you should stop. Principle §29: rollback is a prerequisite, not optional. Validate before starting.
Everything in the prompt, nothing in the repo If the agent does not see it in the repo, it does not exist. Slack, Google Docs, tribal knowledge are invisible. Encode all knowledge in the repository as versioned artifacts.
Orchestrating without a harness More agents without constraints = more drift, faster. High throughput + zero guardrails = exponential tech debt. Tier 2 is a prerequisite for Tier 3. Always.
